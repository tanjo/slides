# 1. なぜキャッシュが必要なのか？

Webアプリケーションのパフォーマンス改善において、**「そもそも通信しない」「そもそも計算しない」**ことが最強のチューニングです。

* **ユーザー体験の向上:** 画面の表示速度が劇的に改善（Core Web Vitals向上）。
* **サーバー負荷の軽減:** Rails（CPU/DB）の処理をスキップできる。
* **通信量の削減:** モバイル回線などでの帯域幅を節約。

キャッシュを制する者は、Webのパフォーマンスを制します。

---

# 2. キャッシュの2大戦略 (1/2)
## ① 鮮度ベース（通信すらしない）

* 「このデータは1時間変わらないよ」とブラウザに教える。
* ブラウザは指定された時間、サーバーに一切リクエストを送らず、手元のデータを使います。

**関連ヘッダー:** `Cache-Control: max-age=3600`

---

# 2. キャッシュの2大戦略 (2/2)
## ② 検証ベース（変更確認のみ）

* 「データを持ってるけど、最新版か確認して！」とサーバーに聞く。
* サーバーは「変わってないよ（304）」または「新しいデータ（200）」を返します。

**関連ヘッダー:** `ETag`, `Last-Modified`

---

# 3. Cache-Control のよくある誤解 (1/2)
## no-cache vs no-store

開発者が最も勘違いしやすいディレクティブ（指示）です。

* ❌ `no-cache` = 「キャッシュを保存してはいけない」
* ⭕️ `no-cache` = **「キャッシュを使ってもいいけど、毎回必ずサーバーに変更がないか検証しなさい」**

* ⭕️ `no-store` = **「絶対にキャッシュを保存するな（機密情報など）」**

---

# 3. Cache-Control のよくある誤解 (2/2)
## ヘッダー指定の早見表

| ヘッダー指定 | 通信する？ | 用途の例 |
| :--- | :--- | :--- |
| `max-age=3600` | 期限内はしない | 頻繁に変わらないマスタデータ |
| `no-cache` | 毎回する(検証のみ) | 常に最新を見せたいAPIレスポンス |
| `no-store` | 毎回する(全取得) | マイページ、決済情報など |

---

# 4. ETag と 304 Not Modified (1/2)
## Railsのデフォルト動作

RailsのAPIモードでは、デフォルトで **ETag（検証ベースのキャッシュ）** が有効になっています。

* レスポンスボディのハッシュ値を `ETag` として付与。
* 内容が変わっていなければ、データ本体を省いて **304 Not Modified** を返します。
* サーバー負荷削減よりも**「通信量の削減」**に寄与します。

---

# 4. ETag と 304 Not Modified (2/2)
## 通信の流れ

1. **TS -> Rails:** `GET /api/users/1`
2. **Rails:** ハッシュ値を作成し返す。
   * `ETag: "W/12345"` / `Status: 200 OK`
3. **TS -> Rails (2回目):** 前回のETag値を入れてリクエスト。
   * `If-None-Match: "W/12345"`
4. **Rails:** ETagを再計算。一致すればデータ本体を省く。
   * `Status: 304 Not Modified`

# 5. CDNとの連携 (1/2)
## s-maxage でRailsを守る

ブラウザとRailsの間にあるCDN（Cloudflare等）にキャッシュさせることで、Railsへのリクエスト自体をブロックできます。

**ブラウザにはキャッシュさせず、CDNにだけさせたい場合:**
`Cache-Control: max-age=0, s-maxage=3600`

* ブラウザ (max-age): 保持しない
* CDN (s-maxage): 1時間保持する

---

# 5. CDNとの連携 (2/2)
## キャッシュのパージ（無効化）

CDNにキャッシュさせた場合、データが更新された際にキャッシュを消す仕組みが別途必要になります。

* **解決策:** Railsのモデルの `after_commit` コールバック等で、CDNのAPIを叩き、特定のパスのキャッシュをパージ（削除）します。

---

# 6. SPAのアセットキャッシュ戦略 (1/2)
## Cache Busting（キャッシュバスティング）

ReactやVueのビルド成果物（JS, CSS）のキャッシングの鉄則です。
ファイル名にハッシュ値を含めることで**「最強のキャッシュ」**と**「即時反映」**を両立させます。

* **ファイル名:** `main-[hash].js` (例: `main-a1b2c3d4.js`)
* **ヘッダー:** `Cache-Control: max-age=31536000, immutable` (1年間)

---

# 6. SPAのアセットキャッシュ戦略 (2/2)
## Cache Bustingの仕組み

1. JSの中身が変わらなければ、ずっとブラウザのキャッシュを使う。
2. デプロイしてJSの中身が変わると、ファイル名が `main-e5f6g7h8.js` に変わる。
3. `index.html` が新しいファイル名を読み込むため、古いキャッシュは無視され最新が読み込まれる。

---

# 7. バックエンドでのキャッシュ (1/2)
## Fragment Caching

HTTPレイヤーだけでなく、Rails内部でもキャッシュを行うことでレスポンス生成を高速化します。

``` ruby
# jbuilderの例 (Redis等に保存)
json.cache! ['v1', @user], expires_in: 1.hour do
  json.extract! @user, :id, :name, :bio
  json.posts @user.posts, :id, :title
end
```

`@user.updated_at` が更新されると、自動でキャッシュが破棄されます。

---

# 7. バックエンドでのキャッシュ (2/2)
## Low-Level Caching

重い計算結果や、外部APIのレスポンスを直接キャッシュしたい場合に使います。

``` ruby
# 12時間キャッシュする例
Rails.cache.fetch("heavy_calc_#{user.id}", expires_in: 12.hours) do
  # 重い集計処理や外部API呼び出し
  AnalyticsService.calculate_score(user)
end
```

---

# 8. まとめ

* **鮮度** と **検証** の違いを理解して使い分ける。
* `no-cache` ≠ `no-store`。機密情報の漏洩に注意。
* APIは **ETag** を意識し、必要に応じて **CDN** で守る。
* SPAのアセットは **Cache Busting** で長期間キャッシュ。
* Rails内部の重い処理は **Redisキャッシュ** で回避する。
